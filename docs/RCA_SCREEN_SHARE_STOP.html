Root Cause Analysis

Issue: Screen Share Stop & Recording State Synchronization

Date: October 26, 2023
Status: Resolved

1. Issue Description

Problem: When a user stopped screen sharing while a recording was active, the recording process did not stop automatically. The user had to manually click "Stop Recording". Additionally, the "Stop Recording" button remained in an active (clickable) state even after the recording had technically stopped, leading to confusion and multiple "Stopped Recording" logs.

Impact:

  • User frustration due to manual intervention required.
  • Potential loss of the last few seconds of recording if the user didn't stop it immediately.
  • Confusing UI state where buttons didn't reflect the actual application status.

2. Root Cause Analysis

2.1. Missing Event Handling for Screen Stop Defect

Root Cause: The application was listening for the ended event on the screen share video track, but the logic inside the event listener only handled cleaning up the screen share UI (stopScreenSharing). It did not check if a MediaRecorder session was currently active.

Code Defect:

// Old code in shareScreen.js
shareStream.getVideoTracks()[0].addEventListener('ended', () => {
    console.log("Screen sharing ended");
    stopScreenSharing(); // Only cleaned up UI, didn't stop recording
});

2.2. Button State Management Defect

Root Cause: The button states (enabled/disabled/colors) were primarily managed by the changeButtons helper function, which only updated CSS classes. It did not set the HTML disabled attribute.

Secondary Cause: The stopRecording function and mediaRecorder.onstop event handler didn't explicitly reset the button states to "Ready to Record" (Start=Enabled, Stop=Disabled).

3. Solution Implementation

3.1. Automated Recording Stop Fix

Fix: We updated src/js/shareScreen.js to check the MediaRecorder state when screen sharing ends.

Logic:

  1. Listen for the ended event on the screen track (triggered by browser UI).
  2. Check if (mediaRecorder.state === 'recording').
  3. If true, call stopRecording() programmatically.
  4. Show a Toast notification to inform the user why the recording stopped.

Code Change:

// New code in shareScreen.js
shareStream.getVideoTracks()[0].addEventListener('ended', () => {
    if (typeof mediaRecorder !== 'undefined' && mediaRecorder.state === 'recording') {
        stopRecording();
        showToast('recording-stopped-toast');
    }
    stopScreenSharing();
});

3.2. UI Feedback (Toast Notification) Fix

Fix: Added a Bootstrap Toast component to index.html and a helper function showToast in scripts.js.

Benefit: Provides clear, non-intrusive feedback that the system took an automatic action.

3.3. Button State Synchronization Fix

Fix: Updated src/js/screenRecorder.js to explicitly manage the disabled attribute of buttons.

Logic:

  • On Start: startBtn.disabled = true, stopBtn.disabled = false.
  • On Stop: startBtn.disabled = false, stopBtn.disabled = true.

Code Change:

// New code in screenRecorder.js (mediaRecorder.onstop)
const startRecordBtn = document.getElementById('start-record');
startRecordBtn.disabled = false; // Re-enable start
startRecordBtn.classList.remove('disabled');

const stopRecordBtn = document.getElementById('stop-record');
stopRecordBtn.disabled = true; // Disable stop
stopRecordBtn.classList.add('disabled');
